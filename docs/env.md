# Environment & local config

## Local config paths

| OS | Path |
|----|------|
| macOS | `~/Library/Application Support/gimble/` |
| Linux | `~/.config/gimble/` |

Those folders hold profiles, local cache, and any credentials the CLI generates or stores.

---

## `chat.env`

For a **private Gimble Cloud** backend, the CLI reads a small env file (often named `chat.env`) with connection settings.

**Example**

```env
GIMBLE_CLOUD_API_BASE=https://chat.gimble.dev
GIMBLE_CLOUD_API_TOKEN=<auto-generated>
GIMBLE_DEVICE_ID=<auto-generated>
```

**Variables**

| Variable | Role |
|----------|------|
| `GIMBLE_CLOUD_API_BASE` | API base URL for your cloud backend |
| `GIMBLE_CLOUD_API_TOKEN` | Auth token (usually created during session/setup and stored locally) |
| `GIMBLE_DEVICE_ID` | Device identifier (usually auto-generated) |

**Good practice**

- Keep `chat.env` out of version control; do not commit secrets.
- Treat `GIMBLE_CLOUD_API_TOKEN` as sensitive; rotate if it leaks.

---

## Logs

The CLI may write **local session logs**; in cloud mode, uploads use **sanitized** data. To inspect files on disk, look under the [config paths](#local-config-paths) above—often under `session` or `logs` subfolders (names may vary by version).

---

## Network / proxy

If outbound HTTPS must go through a proxy, set the usual variables **before** starting Gimble:

```bash
export HTTPS_PROXY="https://proxy.example:3128"
export HTTP_PROXY="http://proxy.example:3128"
```

Adjust host, port, and scheme to match your network. For more symptom-oriented steps, see [Troubleshooting — Network, proxy, or cloud uploads](troubleshooting.md#network-proxy-or-cloud-uploads).
